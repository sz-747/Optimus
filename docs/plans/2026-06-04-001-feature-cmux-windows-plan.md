## 1. Goal

Build a **native** Windows desktop terminal app for managing AI coding agents — a Windows-native re-implementation of cmux. "Native" is the whole point: **no Electron, no Tauri, no Chromium wrapper.** The terminal must be fast and the UI must be a real Windows app.

The app is a multi-pane, multi-workspace terminal. Three value-add features differentiate it from a commodity terminal and are the reason the project exists:

1. **Workspace sidebar** — a vertical-tab list of workspaces, each row showing git branch, PR status, working directory, and the latest notification text.
2. **Notification system** — detect terminal notification escape sequences (OSC 9 / 99 / 777), light up the relevant pane/tab, raise a Windows toast, **and** ship a CLI command so agent hooks can fire notifications.
3. **Split panes** — horizontal and vertical splits.

Everything else (the terminal engine itself) is commodity and exists to support those three.

## 2. Fixed architecture (decided — not up for re-debate)

These were settled before planning and confirmed by research. They are constraints, not options.

| Concern | Technology | Notes |
|---|---|---|
| App shell (windows, sidebar, tabs, menus) | **C# + WinUI 3** (Windows App SDK) | Replaces macOS Swift/AppKit. Unpackaged app, .NET 9, Windows 11 target. |
| Terminal engine (VT parse + grid + GPU render) | **Rust `cdylib`** | Replaces macOS libghostty/Zig. One DLL: `cmux_engine.dll`. |
| C# ↔ Rust boundary | **FFI / P/Invoke** | Bindings generated by **csbindgen**. The fiddly layer — planned explicitly in §6. |
| Running commands | **ConPTY** (Windows pseudoconsole) | Owned by Rust. |
| GPU text rendering | **wgpu (DX12) + glyphon**, composited into a WinUI 3 `SwapChainPanel` | See §5 for the interop path. |
| Optional in-app browser pane | **WebView2** | Phase 6. |

### 2.1 Decisions resolved during grilling (recap)

- **Render model:** Rust renders into a DXGI **composition** swapchain that WinUI 3 hosts via `SwapChainPanel` + `ISwapChainPanelNative`. (Not "C# owns a D3D surface".)
- **Phase 1 acceptance bar:** a *full* single pane — ANSI color, cursor, scrollback, live resize, **plus** mouse selection and clipboard copy. Deferred from Phase 1: font fallback, emoji, ligatures, GPU perf-tuning.
- **ConPTY ownership:** **Rust** owns it (creates the pseudoconsole, spawns the shell, runs the read loop, feeds its VT parser). C# only forwards input + resize.
- **Toolchain:** modern defaults — .NET 9, latest Windows App SDK (1.7.x stable / 1.8 GA), unpackaged, csbindgen, Windows 11.

### 2.2 Decisions refined *by* the research (new — adopted as defaults under "you pick")

1. **Swapchain ownership → wgpu owns it.** wgpu's DX12 backend exposes `wgpu::SurfaceTargetUnsafe::SwapChainPanel(*mut c_void)`, and internally performs `IDXGIFactory2::CreateSwapChainForComposition` **and** `ISwapChainPanelNative::SetSwapChain`. So the plan is: **C# QIs the `SwapChainPanel` for its native interface and passes the raw pointer across FFI; Rust hands it to wgpu; wgpu creates and binds the swapchain.** This is simpler than manual DXGI and removes the hand-rolled `SetSwapChain` dance. (Pin wgpu ≥ 29.x and force `Backends::DX12`.)
   - **Fallback (documented, not default):** raw **D3D11 + DirectWrite/Direct2D**, where Rust calls `CreateSwapChainForComposition` and `ISwapChainPanelNative::SetSwapChain` itself via the `windows` crate. Kept as an escape hatch if the wgpu↔SwapChainPanel arm proves unstable under resize/DPI (it is a real but lightly-traveled code path — spike it first, see §7).
2. **VT core → `wezterm-term`** (pinned to a git rev), not `alacritty_terminal`. Rationale: wezterm's `AlertHandler` trait yields parsed `Alert::ToastNotification { title, body, focus }` for OSC 9 and OSC 777 and `Alert::SetUserVar` for iTerm2 OSC 1337; alacritty's `EventListener`/`Event` enum has **no** notification variants and would force a fork of its private `osc_dispatch`. OSC notifications are a core feature, so this is dispositive. Add a tiny custom OSC-99 (Kitty) sniffer since neither core parses it.
3. **ConPTY → raw Win32 via the `windows` 0.62.x crate**, not the `conpty` crate (which lacks `ResizePseudoConsole` and pins a stale `windows 0.54`).
4. **Glyph stack → `glyphon`** (bundles `cosmic-text` + `swash`) for the GPU glyph atlas + in-pass text renderer; **custom instanced quads** for the cell-background grid, cursor, and selection highlight.
5. **IPC transport → Windows named pipe** (`\\.\pipe\cmux-<variant>`), with the wire protocol (V1 text + V2 JSON, `\n`-framed) ported verbatim from macOS so the CLI and agent-hook contracts carry over. AF_UNIX is a viable alternative but named pipes give first-class ACL-based peer auth on Windows. (Decision is reversible; isolated behind a transport interface.)

**Three processes / artifacts:**

1. **`cmux.exe`** — the WinUI 3 app (C#) + the embedded `cmux_engine.dll` (Rust). One engine instance per terminal surface (pane/tab).
2. **`cmux_engine.dll`** — the Rust engine: VT core, ConPTY, GPU renderer. Pure C ABI.
3. **`cmux` CLI** (`cmux.exe` CLI, separate binary) — talks to the app over the named pipe; drives notifications and sidebar metadata; installs agent hooks.

**Two directions of data flow across FFI:**

- **C# → Rust (calls):** create/destroy engine, attach swapchain panel, spawn shell, send key/text/mouse/scroll, resize (cols/rows + pixels + DPI), render tick.
- **Rust → C# (callback):** a single `HostEvent` callback delivering notifications (OSC 9/99/777), title changes, bell, cwd changes, child-exit, progress — modeled on Ghostty's single `handleAction(target, action)` dispatch.

## 5. The swapchain interop path (the historically scary seam — now de-risked)

This is the single most important technical seam and the thing Phase 1 must prove first.

### 5.1 Chosen path (wgpu owns the swapchain)

1. C# creates a `SwapChainPanel` in XAML inside each terminal pane.
2. C# obtains the panel's native COM interface (`ISwapChainPanelNative`, IID `{63aad0b8-7c24-40ff-85a8-640d944cc325}`, declared in `microsoft.ui.xaml.media.dxinterop.h`) via hand-authored `[GeneratedComInterface]` interop, and gets the underlying `ISwapChainPanel` pointer.
3. C# passes that raw pointer (`nint`) into `cmux_engine_attach_swapchain_panel(engine, panel_ptr)` across FFI.
4. Rust calls `instance.create_surface_unsafe(SurfaceTargetUnsafe::SwapChainPanel(panel_ptr))`. wgpu's DX12 backend internally calls `CreateSwapChainForComposition` and `ISwapChainPanelNative::SetSwapChain`. wgpu refcounts the panel.
5. Rust configures the surface (`surface.configure`) with the physical pixel size and presents on its own render thread.

### 5.2 Threading rules (must be obeyed or it crashes / blurs)

- **`create_surface_unsafe` / the `SetSwapChain` it triggers should occur on the UI thread.** C# marshals the attach call appropriately (the panel pointer is a UI-thread object). Practically: call `attach_swapchain_panel` from the pane's `Loaded` handler (UI thread); Rust may need to hop the actual surface creation — design the ABI so attach can run synchronously on the calling (UI) thread and only the *render loop* runs on the Rust render thread.
- **Present / `surface.configure` (resize) run on the Rust render thread**, off the UI thread, on the engine's own cadence.
- **Device-lost:** `Present`/`configure` can return device-removed; Rust must recreate device + surface and re-attach. No event fires — detect from results.

### 5.3 DPI / resize (most error-prone area)

- Physical pixels = `SwapChainPanel.ActualWidth/Height` (DIPs) × `CompositionScaleX/Y`.
- Subscribe to **both** `SizeChanged` and `CompositionScaleChanged`; forward `(cols, rows, pixel_w, pixel_h, dpi_scale)` to `cmux_engine_resize`.
- With wgpu owning the swapchain, wgpu's `surface.configure(width,height)` handles buffer resize. The inverse-scale matrix (`IDXGISwapChain2::SetMatrixTransform` with `1/scale`) is handled internally by wgpu's composition path; if the fallback raw-D3D path is taken, the plan must apply it manually together with `ResizeBuffers` on the same frame.
- Render glyphs at **physical** resolution (DWrite/atlas DPI = `96 × scale`); never let XAML upscale a 96-DPI buffer (blurry text).

### 5.4 Pitfalls captured for the build

- No first-party WinUI 3 + C# externally-created-swapchain sample exists — the `ISwapChainPanelNative` COM interop is hand-rolled (highest chance of a wrong IID/vtable). Verify against the C++ shape.
- `SwapChainPanel` perf guidance: keep ≤ ~4 live panels; **reuse** panels across panes rather than spawning unbounded ones (matters for splits in Phase 2).
- Don't present while the window is minimized/occluded (`DXGI_STATUS_OCCLUDED` → throttle).
- Gate presents on a "panel alive" flag set from `Loaded`/`Unloaded`; stop the render thread before teardown.

## 6. The FFI contract (Rust C ABI — the central artifact)

Designed for csbindgen: opaque handle pointers, `#[no_mangle] pub extern "C"`, plain ints/pointers, UTF-8 `ptr+len` strings, function-pointer callback with an opaque `user_data`. Modeled on Ghostty's single-dispatch action callback. **This is a specification, not implementation.**

```rust
#[repr(C)] pub struct Engine;          // opaque; C# never dereferences

// ---- lifecycle ----
#[no_mangle] pub extern "C" fn cmux_engine_create(opts: *const EngineOptions) -> *mut Engine;
#[no_mangle] pub extern "C" fn cmux_engine_destroy(engine: *mut Engine);

// ---- renderer attach (C# passes the SwapChainPanel native pointer) ----
#[no_mangle] pub extern "C" fn cmux_engine_attach_swapchain_panel(engine: *mut Engine, panel: *mut c_void) -> i32; // 0 ok, <0 err
#[no_mangle] pub extern "C" fn cmux_engine_detach(engine: *mut Engine);

// ---- pty / shell ----
#[no_mangle] pub extern "C" fn cmux_engine_spawn_shell(engine: *mut Engine,
    cmdline_utf8: *const u8, cmdline_len: usize, cwd_utf8: *const u8, cwd_len: usize) -> i32;

// ---- input ----
#[no_mangle] pub extern "C" fn cmux_engine_send_text(engine: *mut Engine, utf8: *const u8, len: usize);
#[no_mangle] pub extern "C" fn cmux_engine_send_key(engine: *mut Engine, vk: u32, modifiers: u32, down: bool);
#[no_mangle] pub extern "C" fn cmux_engine_send_mouse(engine: *mut Engine, x: f32, y: f32, button: u32, kind: u32, modifiers: u32);
#[no_mangle] pub extern "C" fn cmux_engine_send_scroll(engine: *mut Engine, delta_lines: f32);

// ---- resize (cols/rows + pixels + dpi) ----
#[no_mangle] pub extern "C" fn cmux_engine_resize(engine: *mut Engine, cols: u16, rows: u16,
    pixel_width: u32, pixel_height: u32, dpi_scale: f32) -> i32;

// ---- selection / clipboard (Phase 1 bar) ----
#[no_mangle] pub extern "C" fn cmux_engine_selection_text(engine: *mut Engine, out: *mut ByteBuffer) -> i32;
#[no_mangle] pub extern "C" fn cmux_buffer_free(buf: ByteBuffer);

// ---- host event callback (notifications, title, bell, cwd, child-exit, progress) ----
#[repr(C)] pub struct HostEvent {
    kind: u32,                                   // 0=Toast,1=Title,2=Bell,3=SetUserVar,4=Cwd,5=ChildExit,6=Progress
    title_utf8: *const u8, title_len: usize,     // borrowed; valid only during callback
    body_utf8:  *const u8, body_len:  usize,
    arg0: i64,
}
type HostEventFn = extern "C" fn(user_data: *mut c_void, ev: *const HostEvent);
#[no_mangle] pub extern "C" fn cmux_engine_set_event_callback(engine: *mut Engine, cb: HostEventFn, user_data: *mut c_void);

// ---- error detail ----
#[no_mangle] pub extern "C" fn cmux_last_error_message(out: *mut ByteBuffer) -> i32;
```

**Marshalling rules (binding for the build):**
- **Strings:** UTF-8 `ptr + len` both directions (terminal text is arbitrary bytes — never assume NUL termination). Event strings are **borrowed**, valid only during the callback; C# copies immediately into a `string`.
- **Handles:** `*mut Engine` via `Box::into_raw` / `Box::from_raw`. Opaque to C#.
- **Buffers:** `#[repr(C)] ByteBuffer { ptr, len, cap }` returned by value with a paired `cmux_buffer_free`. Each side frees its own allocations.
- **Callbacks:** function pointer + `user_data` (a pinned C# `GCHandle`). C# side uses `[UnmanagedCallersOnly(CallConvs=[CallConvCdecl])]` static methods (no delegate-lifetime hazard). Inside the callback, **immediately `DispatcherQueue.TryEnqueue`** before touching XAML — events originate on the PTY reader/render thread.
- **Calling convention:** `extern "C"` on Rust; `CallConvCdecl` declared explicitly on the C# `[LibraryImport]`/`[UnmanagedCallersOnly]` side.
- **Panics:** every `extern "C"` entry point wraps its body in `std::panic::catch_unwind`. Never unwind across FFI.
- **Binding generation:** csbindgen `build.rs` emits `app/Interop/NativeMethods.g.cs` with `csharp_dll_name("cmux_engine")`. Verify (via `cargo build -vv`) that no exported function was silently skipped. Prefer regenerating into a checked-in file so diffs show up in review. The generated `[DllImport]` layer is the low level; the hand-authored `EngineHandle` wrapper adds `LibraryImport`/`UnmanagedCallersOnly` callback registration and safety.

**Threading model (whole engine):**
- One **render thread** owns the wgpu device + surface (DX12 surfaces are not freely `Send`). It also owns the `wezterm-term` `Terminal`.
- One **PTY reader thread** does blocking `ReadFile` on the ConPTY output pipe and sends byte chunks over an `mpsc` channel to the render thread, which calls `terminal.advance_bytes(&buf)` and re-renders (coalesced: at most once per frame after a burst).
- The **writer** path (keystrokes/paste) writes to the ConPTY input pipe; `wezterm-term` is constructed with a `Box<dyn Write>` over that pipe so the core's own replies (DA/cursor reports) also flow to the shell.
- C# FFI calls (input/resize) post messages to the render thread; they do not touch wgpu directly.
- `HostEvent`s (from `AlertHandler`, on the render thread after parsing) are routed to the C# callback; C# hops to the UI thread.

## 7. De-risking spikes (run as the very first steps of Phase 1)

These are the smallest experiments that prove the two genuine unknowns. **Spikes 1 and 2 are independent — do them in parallel, first.** If spike 1 fails, pivot to the raw-D3D11 fallback *before* building anything else.

1. **wgpu → SwapChainPanel present.** Minimal WinUI 3 app, one `SwapChainPanel`; C# QIs `ISwapChainPanelNative`, passes the pointer to a tiny Rust cdylib that does `Instance(DX12)` → `create_surface_unsafe(SwapChainPanel(ptr))` → clears to a color → presents. Prove: (a) visible/composited, (b) `surface.configure` resize on `SizeChanged`, (c) correct under `CompositionScaleChanged` (DPI). Pin the exact wgpu version that exports `SwapChainPanel`.
2. **ConPTY round-trip in Rust** (`windows` 0.62). Spawn `pwsh`/`cmd`, write a command to the input pipe, read VT bytes on a dedicated thread, prove clean `ResizePseudoConsole` and clean shutdown (`ClosePseudoConsole` + drain output to EOF before closing handles — not draining can deadlock).
3. **`wezterm-term` git-pin + OSC hook.** Feed a recorded byte stream containing `\e]9;hi\e\\` and `\e]777;notify;Title;Body\e\\`; implement `AlertHandler`; assert `Alert::ToastNotification { title, body, focus }` arrives. Add an OSC-99 sniff test for the custom parser.
4. **glyphon on the bound swapchain.** After spike 1, render one screen of monospaced ANSI-colored text + a block cursor via glyphon + instanced bg quads, driven by a real `wezterm-term` `Screen`.
5. **FFI/callback thread-safety smoke test.** Fire a `HostEvent` from the reader thread through the `extern "C"` callback into C#; confirm safe `DispatcherQueue` marshalling, no use-after-free of borrowed string pointers, `catch_unwind` holds.

## 8. Phased build plan

**Smallest-useful-thing-first.** Phase 1 is the build/stop-for-review boundary: **build Phase 1 only, then stop.** Phases 2–6 are detailed to implementation-ready depth now (per request) but are not to be built until Phase 1 is reviewed.

### Build-order rationale (priority vs dependencies)

The user's value-add *priority* is sidebar (1) → notifications (2) → splits (3). The *build order* differs because of hard dependencies:
- The **sidebar's** git-branch/PR/cwd data is pushed in over the **IPC socket** by shell integration, and its "latest notification text" comes from the **notification store**. So sidebar must come *after* notifications and the CLI/IPC layer.
- **Notifications** get their OSC hook for free from the Phase-1 engine (`AlertHandler`), so the surfacing layer is cheap to add early.
- **Splits** are a self-contained extension of Phase 1.

Resulting order: skeleton → splits → notifications → CLI/IPC → sidebar → polish. Each phase ships something usable.

### Phase 1 — Walking skeleton: one real terminal pane  ⟵ BUILD THIS, THEN STOP

**Goal:** prove the entire stack talks end-to-end — C# WinUI 3 UI + Rust engine + ConPTY + FFI + GPU render — in a single window with one fully working terminal pane.

**Scope (in):** single unpackaged WinUI 3 window; one `SwapChainPanel`; Rust `cmux_engine.dll` owning `wezterm-term` + ConPTY + wgpu/glyphon; full FFI for input/resize/events; ANSI 16/256/truecolor; cursor (block + blink); scrollback; live resize + DPI; mouse drag-selection; clipboard copy.

**Scope (out / deferred):** font fallback, emoji, ligatures, subpixel AA; GPU perf-tuning; tabs; splits; sidebar; notifications surfacing; CLI/IPC; WebView2; packaging.

**Implementation units:**
1. **Spikes 1–2** (§7) — wgpu↔SwapChainPanel present; ConPTY round-trip. Gate: both green, or pivot to D3D11 fallback.
2. **Project scaffolding.** `app/Cmux.App.csproj` (net9.0-windows10.0.x, `UseWinUI`, `WindowsPackageType=None`, `WindowsAppSDKSelfContained=true`, RIDs win-x64/win-arm64); `engine/` Rust workspace (`crate-type=["cdylib"]`); `build/` script: `cargo build` → copy `cmux_engine.dll` to app output (`CopyToOutputDirectory=PreserveNewest`). **Verify `dotnet publish -r win-x64` works on .NET 9 now** (known publish/arch bugs for unpackaged self-contained — flagged risk).
3. **FFI surface + csbindgen** (§6). Stand up `lib.rs` with the full ABI (stubbed), wire `build.rs` csbindgen → `NativeMethods.g.cs`. Hand-author `ISwapChainPanelNative.cs` + `EngineHandle.cs`.
4. **Engine core.** `wezterm-term` (git-pinned) `Terminal` + render thread + PTY reader thread + channels (§6 threading). `AlertHandler` impl that emits `HostEvent`s (even though Phase 1 doesn't surface them — wires the path for spike 5 and Phase 3).
5. **ConPTY** (`windows` 0.62): create pseudoconsole, pipes, `STARTUPINFOEXW` + `PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE`, spawn shell (default `pwsh` → fall back `powershell.exe` → `cmd.exe`), reader thread → `advance_bytes`, `ResizePseudoConsole` on resize, drain-on-shutdown.
6. **Renderer.** wgpu DX12 surface from the panel pointer; `render/grid.rs` instanced background-color + cursor + selection quads; `render/text.rs` glyphon atlas + `TextRenderer`; map `wezterm-term` `Screen` cells → quads at fixed advance width; scrollback viewport offset; damage-tracked present.
7. **Input forwarding.** Make the `SwapChainPanel` focusable (`IsTabStop=true`, programmatic `Focus`). `KeyDown`/`KeyUp` → `cmux_engine_send_key` for control keys (Enter/Tab/Esc/arrows/Fn/Ctrl-combos/Backspace/Delete/Home/End/PgUp/PgDn). `CharacterReceived` → `cmux_engine_send_text` for layout/IME/dead-key-resolved text. Modifiers via `InputKeyboardSource.GetKeyStateForCurrentThread`. Pointer events (`GetCurrentPoint(panel)`, ×CompositionScale → cell) → `cmux_engine_send_mouse`/`send_scroll`; `CapturePointer` during drag-select.
8. **Selection + clipboard.** Drag-select updates `wezterm-term` selection; `cmux_engine_selection_text` returns UTF-8; Ctrl+Shift+C (and right-click copy) writes to the WinUI clipboard.
9. **Resize/DPI loop.** `SizeChanged` + `CompositionScaleChanged` → compute cols/rows + physical px → `cmux_engine_resize` → `surface.configure` + `ResizePseudoConsole`.

**Acceptance criteria (the Phase-1 bar):**
- Launch the app → a shell prompt appears in the pane.
- Run commands (`dir`, `git status`, a TUI like `vim`/`htop`-equivalent) with correct **ANSI color**, **cursor**, and **scrollback**.
- **Live resize** reflows correctly (window resize + DPI change between monitors).
- **Mouse drag-selects** text; **Ctrl+Shift+C copies** it to the Windows clipboard.
- No crash on close (clean engine teardown; render thread stopped before panel disposal).

**Risks (Phase 1):** swapchain interop + threading (spike 1); DPI/scale correctness; .NET 9 unpackaged publish; `CharacterReceived` not firing on the panel (WinUI 3 bug history — validate dead-key/IME early); FFI memory/callback safety. (Full risk table in §9.)

### Phase 2 — Tabs + split panes (value-add #3)

**Goal:** multiple terminal surfaces arranged in a binary split tree with tabs, matching macOS bonsplit behavior.

**Model (ported from bonsplit, reconstructed):** a binary tree where a leaf = a **pane** holding a tab list (each tab = one surface = one engine instance), and an internal node = `{ orientation: H|V, first, second, divider_position }`. Splitting is always binary; nested H/V alternation yields complex layouts. Value identifiers `PaneID`/`TabID`. Controller API: `treeSnapshot()`, `focusedPaneId`, `allPaneIds`, `tabs(inPane)`, `selectedTab(inPane)`, `focusPane`, `selectTab`, `setDividerPosition`, `newTerminalSplit(from, orientation, insertFirst, focus)`.

**Implementation units:**
1. `Splits/SplitTree.cs` — the immutable tree model + a mutable controller; reducer over the event vocabulary: `tab.select`, `tab.close`, `tab.dragStart`, `tab.drop`, `pane.focus`, `pane.drop`, `divider.dragStart`, `focus.moveFocus`.
2. WinUI rendering: nested `Grid` + `GridSplitter` per internal node (orientation → row vs column split); each leaf renders a `TerminalPane` (Phase 1). **Reuse SwapChainPanels** (panel pool) rather than unbounded creation (perf guidance).
3. One engine instance per surface; lifecycle tied to tab create/close (`cmux_engine_create`/`_destroy`).
4. Focus routing: directional `focus.moveFocus`; `focusedPanelId` derived from `controller.focusedPaneId` + `selectedTab(inPane)` (never stored independently).
5. Tab bar UI per pane; keyboard shortcuts (new tab, close, split H/V, navigate).
6. *(Optional / can defer to 2.5)* tab + pane drag-and-drop reordering.

**Acceptance:** open multiple tabs; split a pane H and V; nested splits work; drag dividers resize; focus moves between panes; closing a tab/pane tears down its engine cleanly; layout survives within a session.

**Risks:** SwapChainPanel count/perf (mitigate via reuse); focus management correctness; engine teardown races on close. **Note:** if exact divider math / tab-reorder rules matter, initialize the real `vendor/bonsplit` submodule (see §10) before finalizing this model.

### Phase 3 — Notification system (value-add #2), engine→UI surfacing

**Goal:** detect OSC 9/99/777 notifications and surface them — light up the relevant pane/tab, raise a Windows toast, track unread.

**What's already done:** the Phase-1 engine's `AlertHandler` already parses OSC 9 / 777 into `Alert::ToastNotification { title, body }` and emits a `HostEvent`. Add the OSC-99 (Kitty) sniffer. So this phase is mostly the **C# surfacing layer**, ported from macOS.

**Implementation units (ported from `TerminalNotificationStore`/`Queue`/`Policy`):**
1. `Notifications/TerminalNotification.cs` — record: `{ id, tabId, surfaceId, paneId, title, subtitle, body, createdAt, isRead, paneFlash, clickAction }`. (Engine supplies title+body for OSC; subtitle/id added at this layer, matching macOS where OSC carries only title+body.)
2. `NotificationStore` — derived indexes: `unreadCount`, `unreadCountByTab`, `unreadByTabSurface`, `latestByTab` (the last feeds the sidebar in Phase 5). Unread sets: manual + panel-derived + restored.
3. `NotificationQueue` / mutation bus — coalescing key `(generation, {tabId, surfaceId})`; **max 16 per drain batch**; re-validate the target tab/surface still exists before delivery (drop orphans). Drain on the UI thread.
4. `NotificationPolicyEngine` — effects toggles (`record, markUnread, reorderWorkspace, desktop, sound, command, paneFlash`), patch/merge semantics to rewrite or suppress.
5. UI surfacing: pane/tab flash + unread badge; **Windows toast** via `AppNotificationManager` (Windows App SDK) — note unpackaged apps need COM-activation registration for toast click-through; if that proves heavy for v1, start with in-app pane flash + badge and add OS toast as a sub-step.
6. The `HostEvent` callback (`kind=Toast`) → `DispatcherQueue.TryEnqueue` → `store.add(...)`.

**Acceptance:** an agent (or `printf '\e]777;notify;Title;Body\a'`) in a pane raises an in-app notification; the originating tab/pane flashes and shows an unread badge; a Windows toast appears; marking read/dismiss clears state; bursts coalesce (no storm).

**Risks:** unpackaged toast registration; coalescing/target-revalidation correctness; threading from render thread → UI thread.

### Phase 4 — CLI + IPC named-pipe socket (completes value-add #2; plumbs the sidebar)

**Goal:** a `cmux` CLI and the app-side server that backs it — enabling agent hooks to fire notifications **and** shell integration to push sidebar metadata (git branch, PR, cwd).

**Implementation units (ported from macOS CLI + `TerminalController` socket dispatch):**
1. `Ipc/PipeServer.cs` — named-pipe server (`\\.\pipe\cmux-<variant>`). **Dual protocol:** V1 (space-separated text, `\n`-terminated) + V2 (single-line JSON `{id, method, params}` → `{ok, result|error}`, `\n`-framed). Plus `events.stream` (cursor `after_seq`, frame types `event`/`ack`/`heartbeat`).
2. **Auth/security:** restrict the pipe ACL to the current user SID (Windows analog of macOS's `st_uid==getuid()` check); optional peer verification via `GetNamedPipeClientProcessId`. Three access modes (`allowAll`/`cmuxOnly`/`password`); back the password store with **DPAPI / Windows Credential Manager** (macOS used Keychain).
3. **Socket path / discovery:** env precedence `CMUX_SOCKET_PATH` → legacy `CMUX_SOCKET`; variant-aware discovery (debug/nightly/staging/tagged) with connect-probing.
4. **CLI commands** (port the macOS verb set): `notify --title --subtitle --body --workspace --surface --window` (routing: explicit ws+surface → target; surface/window → `notification.create`; else → `notification.create_for_caller` with TTY/console-handle resolution — on Windows substitute a per-surface env var like `CMUX_SURFACE_ID` injected by shell integration, since there's no TTY); `send`→`surface.send_text`; `send-key`→`surface.send_key`; `list-/dismiss-/mark-/open-/jump-to-` notifications; `report_git_branch`, `report_pr`, `report_pwd`, `report_ports`, `set_status`, `set_progress`, `log`.
5. **Agent hooks:** port the hook installer model — a shell snippet (emit **`.ps1` and `.cmd`** variants) that calls `cmux --socket <pipe> hooks <agent> <subcommand>` for lifecycle points (`session-start`, `prompt-submit`, `stop`, `notification`, `session-end`, `session-finalize`), gated on `$env:CMUX_SURFACE_ID`. Cover the same agents (codex, claude/hermes, gemini, cursor, copilot, …).

**Acceptance:** `cmux notify --title X --body Y` from inside a pane raises a notification in that surface; an agent hook fires a notification on stop; `cmux report_git_branch main --status=dirty` updates app state; auth rejects pipes/peers not owned by the user.

**Risks:** named-pipe ACL/peer-auth correctness; TTY→surface mapping without a real TTY (rely on injected env var); protocol parity with existing hook scripts.

### Phase 5 — Workspace sidebar (value-add #1)

**Goal:** the vertical-tab sidebar — one row per workspace showing git branch, PR status, working directory, latest notification text, and unread state.

**Implementation units (ported from `Workspace` + `SidebarWorkspaceRenderItem`):**
1. `Sidebar/Workspace.cs` — owns per-workspace state: `title/customTitle`, `currentDirectory`, `panels`, `gitBranch: {branch, isDirty}`, `pullRequest: {number, label, url, status: open|merged|closed, branch, isStale}`, `statusEntries`, `progress`, `latestConversationMessage`, unread sets, and the split controller (Phase 2).
2. **Data sourcing (critical):** git branch + PR status are **NOT computed by the app** — they arrive via the Phase-4 socket commands (`report_git_branch`, `report_pr`, `report_pwd`, …) from shell integration / agent hooks. `cwd`/title also come from engine `HostEvent`s (cwd-changed/title). Latest notification text comes from `NotificationStore.latestByTab` (Phase 3). Keep the "watch git status" gate (a default that can suppress reporting).
3. **Snapshot-boundary rule (issue #2586 lesson):** sidebar rows must bind to **immutable value-type DTOs/records** (`SidebarWorkspaceRenderItem` enum: `groupHeader` | `workspace`), **never** to a live `INotifyPropertyChanged`/observable workspace object — doing so caused a 100% CPU spin on macOS. Recompute snapshots on the UI thread; hand `ListView`/`ItemsRepeater` rows value copies.
4. UI: vertical `ListView` with grouping (pinned/groups), per-row branch + PR badge + cwd + latest-notif + unread dot; click selects/focuses the workspace.

**Acceptance:** sidebar lists workspaces; each row shows live git branch, PR status, cwd, and latest notification; unread badges update; selecting a row focuses that workspace; **no CPU spin** (verify rows use DTO snapshots, not observables).

**Risks:** the snapshot-boundary discipline (easy to regress into the CPU-spin bug); keeping derived snapshots in sync without over-rendering.

## 9. Cross-cutting concerns

### 9.1 Riskiest parts (flagged, with mitigations)

| # | Risk | Phase | Mitigation |
|---|---|---|---|
| 1 | **wgpu↔SwapChainPanel interop + threading** (lightly-traveled code path; no first-party C# sample; hand-rolled COM) | 1 | Spike 1 *first*; raw-D3D11+DirectWrite fallback documented; obey UI-thread-attach / render-thread-present split. |
| 2 | **DPI / composition-scale correctness** (blurry or mis-sized text) | 1 | Physical-px sizing = ActualSize×CompositionScale; wgpu handles matrix transform; render glyphs at 96×scale. |
| 3 | **.NET 9 unpackaged self-contained publish** (known community bugs) | 1 | Verify `dotnet publish -r win-x64/arm64` in unit 2 before building features; use WinAppSDK 1.7.x/1.8. |
| 4 | **Keyboard/text correctness** (`CoreWindow` gone; `CharacterReceived` firing bugs; dead-key/IME) | 1 | Panel focusable + KeyDown(control) + CharacterReceived(text); validate dead-key/CJK early; never build text from raw VirtualKey. |
| 5 | **FFI memory/callback safety** (cdecl mismatch, non-blittable bool, cross-allocator free, un-rooted callback) | 1 | `LibraryImport` + `UnmanagedCallersOnly` fn pointers; explicit `CallConvCdecl`; paired `*_free`; `catch_unwind`; DispatcherQueue hop in every callback. |
| 6 | **Unpackaged Windows toast registration** | 3 | Start with in-app flash+badge; add OS toast as a sub-step with COM-activation registration. |
| 7 | **Named-pipe peer auth** (Windows has no `getuid`) | 4 | Pipe ACL → current user SID; optional `GetNamedPipeClientProcessId` verification. |
| 8 | **Sidebar CPU-spin (#2586)** | 5 | Snapshot-boundary rule: DTO/record rows, never observables. |
| 9 | **SwapChainPanel proliferation** with many panes | 2 | Panel pool / reuse; keep live panels low. |

### 9.2 Testing strategy
- **Rust:** unit tests for the VT/OSC hook (spike 3), ConPTY round-trip (spike 2), and `ByteBuffer`/string round-trips. `catch_unwind` boundary tests.
- **FFI:** the callback thread-safety smoke test (spike 5) becomes a permanent test.
- **C#:** port the macOS notification test suite intent — `TerminalNotificationQueueTests` (coalescing, max-16, target-revalidation), `TerminalNotificationSocketActionTests`, `WorkspaceManualUnreadTests`, `TerminalControllerSocketSecurityTests` (access modes/auth). Sidebar snapshot tests asserting rows are value types.
- **Integration:** scripted ConPTY sessions feeding known VT/OSC streams; UI automation for resize/DPI and selection/copy.

### 9.3 Build / CI
- `cargo build --release` → copy `cmux_engine.dll` into the app output (pre-build MSBuild target) for both `win-x64` and `win-arm64`.
- csbindgen regeneration checked in; CI fails if `NativeMethods.g.cs` is stale (diff check).
- `dotnet publish` smoke test per RID on .NET 9 in CI (guards risk #3).
